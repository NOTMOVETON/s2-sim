# S2 — методологический контракт AI-агентов

> Этот документ описывает, как мы работаем над S2. Он отвечает на вопрос «как действовать», а не «что реализовывать».
> Источники истины: `AGENTS.md`, `memory-bank/`, `docs/PROJECT.md`, `docs/ARCHITECTURE.md`, `docs/QUICKSTART.md`, task-файлы в `docs/00-*.md`.

---

## 1. Роль документа

### Зачем

- Чтобы любой новый агент мог продолжить работу без потери контекста.
- Чтобы правила работы не расползались по чату и не терялись между сессиями.
- Чтобы методология была фиксированной, а не менялась по ходу работы.

### Что это значит

- `AGENTS.md` — это рабочий контракт и методология, а не backlog задач.
- Подробная архитектура живёт в `docs/ARCHITECTURE.md`.
- Краткий обзор проекта живёт в `docs/PROJECT.md`.
- Текущая память живёт в `memory-bank/`.
- Текущая задача живёт в task-файле в `docs/`.

---

## 2. Методология работы

### Маленькие, но осмысленные шаги

Каждый шаг — это не просто «написать строчку», а законченный кусок, который:

- имеет понятный смысл в контексте системы;
- можно запустить и проверить;
- можно понять через неделю без дополнительного контекста.

### От малого к большому

- Не начинай с большого куска, если можно начать с маленького.
- Каждый шаг расширяет систему, не ломая предыдущее.
- Не забегай вперёд. Читай docs, делай текущий шаг, проверяй, иди дальше.

### Чистовая реализация, а не временные костыли

- Если уже видно правильное решение, делай сразу чистовую версию.
- Не упрощай реализацию ради скорости, если это ухудшает читаемость, поддержку или предсказуемость.
- Лучше сделать больше, но сделать правильно, если это экономит будущие проблемы.
- Не оставляй «на потом» то, что уже очевидно нужно в финальной форме.

### Критерий качества

- Код должен быть понятным без дополнительного контекста.
- Каждый change set должен быть достаточно маленьким, чтобы его можно было понять и проверить.
- Любое поведение, которое можно покрыть тестом, должно быть проверено тестом.
- Если временное решение создаёт лишнюю сложность, предпочти более правильный вариант сразу.

## 3. Язык и коммуникация

### Зачем

Один язык снижает шум в контексте, упрощает передачу работы между агентами и уменьшает вероятность потери важной информации в хаосе англоязычных комментариев и русскоязычной переписки.

### Правила

- Все ответы, комментарии, вопросы, пояснения и документация — **на русском**.
- Если комментарий в коде действительно нужен, он тоже пишется по-русски.
- Технические термины и имена API не переводить насильно, если это ломает совместимость или стиль проекта.
- Если приходится выбирать между краткостью и ясностью, выбирать ясность.
- Никаких эмодзи в коде и документации, если это не часть пользовательского интерфейса.

## 4. Работа в Docker

### Зачем

Чтобы сборка, тесты и поведение проекта были воспроизводимыми на любой машине, без зависимости от локального окружения.

### Правила

- Любая сборка, тесты, запуск и проверка происходят **в Docker-контейнере**.
- Локальная IDE и хост-машина могут быть пустыми — это нормальная ситуация.
- Не опираться на локально установленные пакеты как на источник истины.
- Если кажется, что зависимости не хватает, сначала проверить `Dockerfile`, `docker-compose` и `CMake` внутри контейнера.
- Если что-то не находится в контейнере, сначала проверить окружение, а уже потом делать вывод о нехватке зависимости.
- `docker compose up` должен собрать и запустить проект.

### Команды

Все команды из корня проекта (`docker/` не обязателен — compose находит контекст):

```bash
# Сборка
docker compose --project-directory docker up --build build          # Собрать проект

# Тесты
docker compose --project-directory docker up --build tests           # Запустить тесты

# Запуск симуляции
docker compose --project-directory docker down                       # Остановить контейнеры
docker compose --project-directory docker up --build sim             # Собрать и запустить

# Альтернатива
cd docker && docker compose down          # Остановить и удалить контейнеры
cd docker && docker compose up sim        # Остановить и запустить (без пересборки)
cd docker && docker compose up --build sim  # С пересборкой

docker compose --project-directory docker logs sim       # Логи симуляции
docker compose --project-directory docker logs -f sim    # Логи в реальном времени
docker ps -a                                            # Статус всех контейнеров
```

### URL после запуска

- Визуализатор: `http://localhost:1937`
- По умолчанию запускается сцена с двумя роботами (`test_two_robots.yaml`)

### Если сайт не открывается

```bash
docker ps -a                                      # Проверить статус всех контейнеров
docker logs docker-sim-1                          # Посмотреть логи симуляции
docker compose --project-directory docker down    # Полностью остановить
docker compose --project-directory docker up --build sim    # Пересобрать и запустить
```

## 5. Память между сессиями и агентами

### Зачем

Проект не должен зависеть от памяти конкретного чата или конкретного агента. Документация и Memory Bank — единственные источники истины.

### Правила

- В начале каждой задачи **прочитать все файлы `memory-bank/*.md`**.
- Считать Memory Bank обязательной памятью проекта.
- Обновлять `memory-bank/` после значимых изменений, принятых решений и закрытых вопросов.
- Если контекста становится слишком много, сокращать активный контекст и переносить устойчивые знания в Memory Bank.
- Если информации не хватает, сначала проверить Memory Bank и проектные docs, а уже потом задавать вопрос.
- Если вопрос или решение уже обсуждены и подтверждены, перенести их из переписки в документацию.

### Когда обновлять Memory Bank

- После завершения значимого шага.
- После принятия архитектурного решения.
- После закрытия открытого вопроса.
- После обнаружения нового паттерна в проекте.

## 6. Структура Memory Bank

Memory Bank — это набор файлов в `memory-bank/`, каждый из которых имеет свою ответственность.

### Файлы

- **`projectbrief.md`** — цели, границы и базовые требования проекта.
- **`productContext.md`** — зачем проект существует и какую проблему решает.
- **`activeContext.md`** — текущая работа, активные решения и ближайшие шаги.
- **`systemPatterns.md`** — архитектурные паттерны, связи компонентов и критические пути.
- **`techContext.md`** — стек, Docker-процедуры, зависимости и ограничения среды.
- **`progress.md`** — что работает, что осталось, известные проблемы и эволюция решений.

### Добавление новых файлов

При необходимости можно добавлять дополнительные файлы в `memory-bank/`, если они помогают держать контекст чище. Например:

- Сложная фича с собственной логикой.
- Спецификация интеграции.
- Процедура деплоя.

## 7. Подагенты и deep planning

### Зачем

Параллелить исследование и сокращать время на анализ без потери качества.

### Правила

- Подагентов можно и нужно использовать для независимого чтения, анализа, сверки фактов и черновиков.
- Deep planning допустим для крупных, неоднозначных или рискованных изменений.
- Подагенты работают по тем же правилам: русский язык, Docker-first, обновление документации и Memory Bank.
- Итоговую архитектурную линию удерживает основной агент.
- Решения подагентов не становятся финальными, пока основной агент не подтвердил и не задокументировал их.

---

## 8. Документация как часть работы

### Зачем

Документация — это живая память проекта, а не архив.

### Правила

- После каждого осмысленного шага обновлять релевантные docs и Memory Bank.
- Фиксировать не только код, но и то, что уже работает, что не реализовано, что решено и что остаётся открытым.
- Любой закрытый вопрос должен быть перенесён из переписки в документацию.
- Если ответ на вопрос влияет на поведение проекта, он должен быть виден и в docs, и в Memory Bank.
- Если документация расходится с реальностью, сначала исправить документацию.

---

## 9. Принятие решений

### Зачем

Чтобы не гадать и не тащить в проект скрытые допущения.

### Правила

- Если требование неоднозначно, спросить пользователя.
- Если есть несколько валидных путей, кратко описать trade-offs и выбрать тот, который лучше соответствует архитектуре.
- Если контекст начал шуметь, сократить его через Memory Bank вместо того, чтобы тащить всё в чат.
- Если задача крупная или неоднозначная, сначала использовать подагентов или deep planning, а уже потом менять код.
- Не принимать архитектурные решения самостоятельно, если есть `[СПРОСИТЬ]` в task-файле.

---

## 10. Что AGENTS.md не должен содержать

### Зачем

Чтобы документ оставался стабильным и пригодным для постоянного использования.

### Правила

- Не превращать его в backlog задач.
- Не копировать сюда полную архитектуру.
- Не расписывать пошаговую реализацию конкретной фичи.
- Не дублировать то, что уже должно жить в `docs/` и `memory-bank/`.

---

## 11. Рабочий цикл задачи

Каждая задача — это цикл:

1. Прочитать `memory-bank/` и relevant docs.
2. Прочитать task-файл целиком.
3. Выполнить минимальный осмысленный шаг.
4. Написать тесты, которые проверяют шаг.
5. Запустить в Docker и убедиться, что всё проходит.
6. Обновить документацию и Memory Bank.
7. Показать результат.

Не пытаться реализовать всю систему разом. Маленький шаг → проверка → следующий шаг.

---

## 12. Обязательное обновление Memory Bank

После каждого завершённого шага **обязательно** обновить следующие файлы:

### Всегда обновлять
- **`memory-bank/progress.md`** — что сделано, что осталось, известные проблемы.
- **`memory-bank/activeContext.md`** — текущая работа, активные решения, ближайшие шаги.

### Обновлять при необходимости
- **`memory-bank/techContext.md`** — если изменились зависимости, стек, Docker-процедуры, ограничения среды.
- **`memory-bank/systemPatterns.md`** — если появились новые архитектурные паттерны, связи компонентов, критические пути.
- **`memory-bank/productContext.md`** — если изменились цели, границы проекта или решённые проблемы.
- **`memory-bank/projectbrief.md`** — если изменились базовые требования или границы проекта.

### Критерий завершения шага

Шаг считается завершённым только после того, как:
1. Все тесты проходят.
2. Memory Bank обновлён.
3. Результат показан пользователю.
