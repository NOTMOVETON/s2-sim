# Задача 04.8 — Kinematic Tree

> **Предыдущий шаг:** `04.7-plugin-fixes.md`

## Цель

Добавить поддержку кинематического дерева агента и интегрировать его в транспортный слой.

**Что это даёт:**
- Сенсоры (GNSS, IMU, Lidar) могут быть смонтированы не в центре робота, а на конкретном звене
- TF-дерево в ROS2 публикует полное дерево фреймов: `base_link → sensor_link → ...`
- Статические трансформы (fixed joints) публикуются один раз; динамические (revolute/prismatic) — каждый кадр
- Транспортный адаптер не знает о KinematicTree — получает уже сплющенный список FrameTransform

**Ограничения задачи:**
- Только YAML-конфиг (URDF — отдельная задача)
- Плагины-актуаторы для управления джоинтами (JointVelocityPlugin) — отдельная задача
- Существующие сенсорные плагины (gnss, imu) пока не меняют поведение: по-прежнему используют `agent.world_pose` для вычислений, но публикуют свой TF-фрейм

---

## Архитектура

### Вариант 2+3: сплющивание дерева + разделение статика/динамика

```
KinematicTree (агент)
  └─ SimTransportBridge::init()
       ├─ Фиксированные джоинты → adapter->register_static_transforms()
       └─ Движущиеся джоинты → AgentSensorFrame::dynamic_transforms (каждый тик)

IAgentPlugin::mount_frame()
  └─ SimTransportBridge::init() — статическое смещение сенсора
       └─ adapter->register_static_transforms()
```

Транспортный адаптер (`ITransportAdapter`) никогда не видит `KinematicTree` — только `std::vector<FrameTransform>`.

## Новые типы

### `FrameTransform` (в `transport_adapter.hpp`)

```cpp
struct FrameTransform {
    std::string parent_frame;   // "base_link", "arm_link_1", ...
    std::string child_frame;    // "lidar_front", "imu_link", ...
    Pose3D      relative_pose;  // смещение ребёнка относительно родителя
};
```

### `KinematicTree` (новый файл `kinematic_tree.hpp`)

```cpp
namespace s2 {

enum class JointType { FIXED, REVOLUTE, PRISMATIC, CONTINUOUS };

struct Joint {
    JointType type{JointType::FIXED};
    Vec3      axis{0.0, 0.0, 1.0};   // ось вращения/трансляции
    double    value{0.0};            // текущий угол (рад) или смещение (м)
    double    min{-M_PI};
    double    max{ M_PI};
};

struct Link {
    std::string name;           // уникальное имя линка
    std::string parent;         // "" для корневого (base_link)
    Pose3D      origin;         // смещение от родителя при value == 0
    Joint       joint;          // джоинт, соединяющий с родителем
};

class KinematicTree {
public:
    void        add_link(Link link);
    bool        empty() const;
    const std::vector<Link>& links() const;

    // Установить текущее значение джоинта
    void set_joint_value(const std::string& link_name, double value);

    // Получить позу линка в мировых координатах (base_pose = поза base_link)
    Pose3D compute_world_pose(const std::string& link_name,
                              const Pose3D& base_pose) const;

    // Собрать список трансформов (для транспорта)
    // Статические (FIXED джоинты): is_static = true
    // Динамические (REVOLUTE/PRISMATIC): is_static = false, current value applied
    void collect_transforms(std::vector<FrameTransform>& out_static,
                            std::vector<FrameTransform>& out_dynamic) const;

private:
    std::vector<Link> links_;
    std::unordered_map<std::string, size_t> link_index_;
};

} // namespace s2
```

## Изменения в SimTransportBridge

### `init()` — собрать и зарегистрировать статические трансформы

После `register_agent(...)`:

```cpp
// Собрать static transforms из KinematicTree агента
std::vector<FrameTransform> static_tfs;
if (agent.kinematic_tree) {
    std::vector<FrameTransform> dyn; // игнорируем при init
    agent.kinematic_tree->collect_transforms(static_tfs, dyn);
}

// Собрать static transforms из mount_frame() плагинов
for (const auto& plugin : agent.plugins) {
    auto mf = plugin->mount_frame();
    if (mf.has_value()) {
        static_tfs.push_back(mf.value());
    }
}

if (!static_tfs.empty()) {
    adapter_->register_static_transforms(agent.id, agent.domain_id, static_tfs);
}
```

### `on_post_tick()` — добавить dynamic_transforms в frame

```cpp
// Динамические трансформы из KinematicTree
if (agent.kinematic_tree) {
    std::vector<FrameTransform> stat; // уже зарегистрированы
    agent.kinematic_tree->collect_transforms(stat, frame.dynamic_transforms);
}
```

## Парсинг YAML (SceneLoader)

Добавить поддержку `links:` в конфиге агента:

```yaml
agents:
  - name: robot_0
    pose: {x: 0, y: 0, z: 0, yaw: 0}
    plugins:
      - type: gnss
        mount: {x: 0.1, y: 0, z: 0.3}   # смещение от base_link
      - type: imu
        mount: {x: 0, y: 0, z: 0.1}
    links:
      - name: arm_link_1
        parent: base_link
        origin: {x: 0.3, y: 0, z: 0.1}
        joint: {type: revolute, axis: [0, 0, 1]}
      - name: camera_link
        parent: arm_link_1
        origin: {x: 0.1, y: 0, z: 0}
        joint: {type: fixed}
```

**SceneLoader изменения:**
- При парсинге плагина: если есть `mount:` — вызвать новый метод на плагине (или поставить через базовый класс)
- Для `links:` создать `KinematicTree` и присвоить `agent.kinematic_tree`

Поскольку `IAgentPlugin::mount_frame()` должен возвращать `FrameTransform`, плагины должны уметь его сохранять. Базовый класс `IAgentPlugin` хранит `mount_frame_` как поле, `from_config()` вызывает `set_mount_frame()` при наличии `mount:` в YAML.

Альтернатива — `SceneLoader` сам читает `mount:` и вызывает `plugin->set_mount_pose(Pose3D)`.

**Выбираем второй вариант**: `SceneLoader` отвечает за монтаж, плагин не знает про TF.

В `IAgentPlugin`:
```cpp
void set_mount_pose(const Pose3D& pose) { mount_pose_ = pose; }
const Pose3D& get_mount_pose() const    { return mount_pose_; }
bool has_mount_pose() const             { return mount_pose_ != Pose3D{}; }

std::optional<FrameTransform> mount_frame() const override {
    if (!has_mount_pose()) return std::nullopt;
    std::string child = sensor_name_.empty()
        ? type() + "_link"
        : sensor_name_ + "_" + type() + "_link";
    return FrameTransform{"base_link", child, mount_pose_};
}
```

`mount_pose_` хранится в базовом классе `IAgentPlugin`. Производные классы не нужно менять.

## Что НЕ входит в эту задачу

- URDF парсер (отдельная задача)

- Изменение вычислений в GnssPlugin/ImuPlugin — они продолжают использовать `agent.world_pose` (sensor данные считаются в центре агента). Кинематическое дерево на этом этапе влияет только на TF-публикацию.
