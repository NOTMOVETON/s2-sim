# Задача 04.9 — URDF Loader + JointVelPlugin + Topic/Rate Config + Kinematic Frames Viz

## Цель

1. Загружать кинематическое дерево агента из URDF-файла
2. Управлять джоинтами (arm, bucket и т.д.) через ROS2 топик
3. Сделать имя топика и частоту публикации конфигурируемыми для каждого плагина
4. Отображать все фреймы кинематического дерева в веб-визуализаторе

## Контекст

Задача 04.8 добавила `KinematicTree` и его интеграцию в транспортный слой. Теперь нужно:
- Заполнять дерево из URDF (а не только из YAML `links:`)
- Управлять динамическими джоинтами из ROS2
- Показывать фреймы дерева в Three.js визуализаторе

## Как работают фреймы и управление

`/cmd_vel` → `DiffDrivePlugin` → `agent.linear/angular_velocity` → SimEngine интегрирует
в `agent.world_pose` → транспорт публикует `odom → base_link` TF.

Всё дерево TF (`base_link → arm`, `base_link → gps_left_link`, ...) висит под `base_link`.
Когда `base_link` движется — все дочерние фреймы переезжают автоматически (TF2).

`/cmd_vel_mount` → `JointVelPlugin` → `agent.kinematic_tree->set_joint_value()` →
`dynamic_transforms` в `AgentSensorFrame` → публикуются в TF каждый тик.

## Изменения A: Конфигурируемый топик и частота плагина

### Проблема

Сейчас топик сенсора = `/<type>/<name>/<suffix>` — захардкожено.
Частота — у каждого плагина своя переменная, нет единого способа переопределить из YAML.

### Решение

**`s2_core/include/s2/plugin_base.hpp`** — добавить в `IAgentPlugin`:
```cpp
// Переопределить имя выходного топика (если пустой — используется конвенция)
const std::string& output_topic() const { return output_topic_; }
void set_output_topic(const std::string& t) { output_topic_ = t; }

// Переопределить частоту из YAML (побеждает дефолт плагина)
void set_base_rate(double hz) { base_rate_hz_ = hz; }
double publish_rate_hz() const override {
    return base_rate_hz_ > 0.0 ? base_rate_hz_ : default_publish_rate_hz();
}
virtual double default_publish_rate_hz() const { return 0.0; }

private:
std::string output_topic_;
double base_rate_hz_{0.0};
```

**`s2_core/include/s2/transport_adapter.hpp`** — в `SensorRegistration`:
```cpp
std::string topic_override;  // если непустой — адаптер использует его напрямую
```

**`s2_core/include/s2/scene_loader.hpp`** — при парсинге плагина ДО `from_config()`:
```cpp
if (node["topic"])           plugin->set_output_topic(node["topic"].as<std::string>());
if (node["publish_rate_hz"]) plugin->set_base_rate(node["publish_rate_hz"].as<double>());
```

**`s2_transport/src/sim_transport_bridge.cpp`** — при создании `SensorRegistration`:
```cpp
reg.topic_override = plugin->output_topic();
```

**`s2_transport/src/ros2_transport_adapter.cpp`** — в `register_sensor()`:
```cpp
std::string topic = reg.topic_override.empty()
    ? make_sensor_topic(reg.sensor_type, reg.sensor_name, suffix)
    : reg.topic_override;
```

**`s2_plugins/include/s2/plugins/gnss.hpp`** — адаптировать под новую схему:
```cpp
double default_publish_rate_hz() const override { return publish_rate_hz_; }
// Убрать переопределение publish_rate_hz()
```

**Пример конфига:**
```yaml
plugins:
  - type: gnss
    name: right
    topic: /gps/right/fix      # вместо /gnss/right/fix
    publish_rate_hz: 5.0       # вместо дефолтных 10 Гц
```

## Изменения C: URDF Loader

### Файлы
- `s2_core/include/s2/urdf_loader.hpp`
- `s2_core/src/urdf_loader.cpp`

### API
```cpp
namespace s2 {
    // Строит KinematicTree из URDF, начиная с root_frame (BFS вниз по дереву).
    // Звенья выше root_frame (например base_footprint) игнорируются.
    // Это предотвращает дублирование TF: мы публикуем odom→base_link сами.
    KinematicTree load_urdf(const std::string& path,
                            const std::string& root_frame = "base_link");
}
```

### Алгоритм

1. Парсим все `<joint>` в `map<string, JointDef>` (`child_link` → {type, parent, origin, axis, limits})
2. Добавляем `root_frame` как корневой Link (parent = "")
3. BFS: для каждого добавленного link ищем joint где `parent_link == link.name` → добавляем child
4. Маппинг типов: `"fixed"→FIXED`, `"revolute"→REVOLUTE`, `"continuous"→CONTINUOUS`, `"prismatic"→PRISMATIC`
5. `<origin xyz="x y z" rpy="r p y">` → `Pose3D{x, y, z, r, p, y}`
6. `<axis xyz="x y z">` → `Vec3{x, y, z}`
7. `<limit lower=".." upper="..">` → `joint.min`, `joint.max`

### Результат для dozer.urdf

Static (FIXED): `gps_right_link`, `gps_left_link`, `gps_middle_link`

Dynamic (CONTINUOUS): `FR_wheel`, `FL_wheel`, `BR_wheel`, `BL_wheel`, `MR_wheel`, `ML_wheel`,
`FMR_wheel`, `FML_wheel`, `BMR_wheel`, `BML_wheel`

Dynamic (REVOLUTE): `arm`, `bucket`

### Интеграция в SceneLoader

```yaml
agents:
  - name: dozer
    domain_id: 50
    urdf: robots/dozer/robot.urdf   # путь относительно конфига
    plugins:
      - type: diff_drive
      - type: joint_vel
        topic: /cmd_vel_mount
        joints:
          - name: arm
            axis: linear_x
            max_vel: 0.01
          - name: bucket
            axis: angular_z
            max_vel: 0.01
```

`urdf:` имеет приоритет над `links:` если заданы оба.

### Rviz2

Наш симулятор публикует все TF (static + dynamic). Геометрия URDF отображается
через стандартный `robot_state_publisher`:
```bash
ros2 run robot_state_publisher robot_state_publisher \
  --ros-args -p robot_description:="$(cat robot.urdf)"
# В rviz2: Add → RobotModel
```

## Изменения E: JointVelPlugin

### Файлы
- `s2_plugins/include/s2/plugins/joint_vel.hpp`
- `s2_plugins/src/joint_vel.cpp`

### Интерфейс

```cpp
class JointVelPlugin : public IAgentPlugin {
public:
    std::string type() const override { return "joint_vel"; }
    std::vector<std::string> command_topics() const override { return {topic_}; }
    bool has_inputs() const override { return true; }

    void from_config(const YAML::Node& node) override;
    // Читает: topic, joints[].{name, axis, max_vel}

    void handle_input(const std::string& json) override;
    // json = Twist: {"linear":{"x":0.0,"y":0.0,"z":0.0},"angular":{"x":0.0,"y":0.0,"z":0.0}}
    // → извлекает нужный axis для каждого joint → сохраняет target_vel

    void update(double dt, Agent& agent) override;
    // Для каждого joint:
    //   new_val = old_val + target_vel * dt
    //   agent.kinematic_tree->set_joint_value(name, new_val)  // с clamping

    std::string to_json() const override;
    // {"plugin":"joint_vel","joints":[{"name":"arm","value":0.3},{"name":"bucket","value":0.1}]}

private:
    std::string topic_{"/cmd_vel_mount"};
    struct JointMapping {
        std::string joint_name;
        std::string twist_axis;  // "linear_x", "linear_y", "linear_z",
                                  // "angular_x", "angular_y", "angular_z"
        double max_vel{0.1};
        double target_vel{0.0};
    };
    std::vector<JointMapping> joints_;
};
```

### Маппинг Twist-полей

| `axis` в YAML | Поле Twist |
|--------------|-----------|
| `linear_x` | `linear.x` |
| `linear_y` | `linear.y` |
| `linear_z` | `linear.z` |
| `angular_x` | `angular.x` |
| `angular_y` | `angular.y` |
| `angular_z` | `angular.z` |

### Управление из ROS2

```bash
# Поднять стрелу со скоростью 0.005 рад/с
ros2 topic pub /cmd_vel_mount geometry_msgs/msg/Twist \
  "{linear: {x: 0.005, y: 0.0, z: 0.0}, angular: {x: 0.0, y: 0.0, z: 0.0}}"
```
