## Предпосылка

После добавления `joint_vel` и управления через веб-интерфейс остались две проблемы на сцене dozer:

1. При поворотах базы через UI робот визуально поворачивается некорректно.
2. Управление скоростями фреймов через `joint_vel` из UI не приводит к движению нужных звеньев.

## Проблема

Текущее поведение нарушает две ключевые гарантии:

- Поворот базы в UI должен быть согласован с математикой симулятора и TF.
- Команды `joint_vel` из UI должны детерминированно изменять значения джоинтов дерева кинематики.

Сейчас обе гарантии не соблюдаются.

## Цель задачи

Восстановить корректную и согласованную работу:

1. Поворота базы (UI ↔ симулятор ↔ TF ↔ визуализация).
2. Управления `joint_vel` (UI ↔ plugin input ↔ `KinematicTree`).

## Наблюдаемые симптомы

### Симптом A: некорректный поворот базы

- При вращении робота в UI направление и/или знак поворота не совпадает с ожидаемым.

- Возможная причина: несоответствие имён джоинтов в сцене фактическим именам звеньев URDF.

## Ожидаемое поведение

## Технические гипотезы (подтверждены)

1. **Подтверждено.** В `app.js` нарушена консистентность ориентации кинематических фреймов.
   - Базовый меш: `mesh.rotation.y = pose.yaw` — корректно (только yaw, roll=0, pitch=0).
   - Кинематические фреймы и link meshes: `rotation.set(roll, yaw, pitch)` с дефолтным порядком XYZ — некорректно.
   - Правильное преобразование Z-up → Y-up: sim ZYX = `Rz(yaw)*Ry(pitch)*Rx(roll)` → Three.js YZX = `Ry(yaw)*Rz(-pitch)*Rx(roll)`. Ось Y симулятора отображается в ось -Z Three.js, что инвертирует знак pitch.
2. **Подтверждено.** В dozer-конфиге для `joint_vel` использовались `arm_joint`, `bucket_joint` вместо `arm`, `bucket`.
3. **Подтверждено.** Регрессия не была покрыта тестом.

## Реализованные исправления

### Симптом A: некорректная ориентация кинематических фреймов

**Файл:** `workspace/s2_visualizer/web/js/app.js`

Функции `updateOrCreateKinematicFrame` (строка ~361) и `updateOrCreateLinkMesh` (строка ~403).

**Было:**
```js
axes.rotation.set(pose.roll || 0, pose.yaw || 0, pose.pitch || 0);
// порядок XYZ (дефолт): Rx(roll)*Ry(yaw)*Rz(pitch) — неверно
```

**Стало:**
```js
axes.rotation.set(pose.roll || 0, pose.yaw || 0, -(pose.pitch || 0), 'YZX');
// порядок YZX: Ry(yaw)*Rz(-pitch)*Rx(roll) — математически корректное преобразование Z-up→Y-up
```

**Математическое обоснование:**
- Позиционная конвертация: `(sim_x, sim_y, sim_z) → (sim_x, sim_z, -sim_y)` (Three.js Y-up)
- Матрица смены базиса M: `sim+X→+X, sim+Y→-Z, sim+Z→+Y`
- `M * Rz(yaw) * M^T = Ry(yaw)` — знак сохраняется ✓
- `M * Ry(pitch) * M^T = Rz(-pitch)` — знак инвертируется ✓
- `M * Rx(roll) * M^T = Rx(roll)` — знак сохраняется ✓

На простых агентах без URDF (roll=0, pitch=0) ошибка не проявлялась — только yaw.
На URDF-моделях с нетривиальными осями суставов ориентация фреймов была неверной.

### Симптом B: не работало управление joint_vel

**Файл:** `workspace/s2_config/scenes/test_dozer.yaml`

`KinematicTree` хранит звенья по именам link-ов (не joint-ов).
Заменены идентификаторы: `arm_joint` → `arm`, `bucket_joint` → `bucket`.
