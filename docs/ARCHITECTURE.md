# Часть I — Цели и философия

## 1. Задача и контекст

### 1.1 Что мы делаем [РЕШЕНО]

S2 — **semantic / high-level interaction simulator** для тестирования высокоуровневой робототехнической логики. Не физический движок общего назначения, не замена Gazebo/Isaac Sim.

Система нужна для:

- тестирования миссий и orchestration,
- интеграции с высокоуровневым стеком робота (Nav2, MoveIt2, custom),
- моделирования дверей, лифтов, людей, триггеров, маркеров, зон и событий,
- масштабирования до десятков-сотен роботов в одной сцене.

### 1.2 Контекст

Есть флот роботов. Каждый робот — полноценный софтовый стек, который принимает миссии от сервера и выполняет их автономно. Сервер управляет флотом: распределяет задачи, контролирует исполнение, перепланирует (по модели OpenRMF).

Текущие решения (Isaac Sim + ROS Control) позволяют максимум 3 робота из-за тяжести симуляции. Но для тестирования fleet management нужны десятки.

### 1.3 Ключевое упрощение [РЕШЕНО]

Всё взаимодействие робота с миром сводится к высокоуровневым фактам:

- Робот не «видит ArUco маркер камерой» — он «оказался рядом с маркером и получил его позицию»
- Робот не «нажимает кнопку двери» — он «отправляет команду и ждёт смены состояния»
- Робот не «физически толкает ящик» — он «отправляет команду захвата и ящик прикрепляется»

Это отсекает: GPU-рендер, contact solver, фотореалистичную перцепцию, камеры.
Позволяет строить мир вокруг состояний, зон, правил, ограничений и событий.

### 1.4 Что S2 делает

- Принимает управляющие сигналы от стека каждого робота через транспортный плагин
- Обновляет состояние всех сущностей каждый тик (кинематика, без физики)
- Применяет эффекты среды к движению и сенсорам
- Вычисляет эффективные ограничения из нескольких источников (resolver)
- Публикует сенсорные данные каждому роботу (GPS, лидар, батарея и др.)
- Содержит интерактивные объекты (двери, лифты, маркеры) с которыми роботы взаимодействуют
- Содержит динамические препятствия (пешеходы)
- Позволяет оператору наблюдать и вмешиваться через визуализатор
- Выполняет raycast для лидарных модулей
- Масштабируется до 100 агентов

### 1.5 Что S2 не делает

- Нет физической симуляции (трение, инерция, столкновения с импульсом)
- Нет рендера изображений
- Нет привязки к ROS (ROS — один из транспортных адаптеров)
- Нет распознавания образов, камер, компьютерного зрения
- Нет контактной механики захвата

### 1.6 Почему не существующие решения

| Инструмент | Проблема |
|-----------|---------|
| Isaac Sim | Требует GPU, избыточен (полная физика, рендер), максимум 3 робота |
| Gazebo | Тяжёлый, медленно масштабируется, сложно кастомизировать |
| Webots | Те же проблемы, чуть легче |
| Stage/flatland | Только 2D, нет интерактивных объектов |

## 2. Принципы проектирования

### 2.1 Ядро однопоточное и authority-based [РЕШЕНО]

Симуляционное ядро — единственный источник истины о состоянии мира. Один поток, детерминированный порядок обновлений. Визуализатор и транспорт — потребители, не владеют состоянием.

Почему: воспроизводимость, понятный порядок событий, простая отладка, нет race conditions.

### 2.2 ROS не является core-зависимостью [РЕШЕНО]

ROS2 — только один из transport adapters. Ядро и визуализатор живут независимо от ROS. ROS topic/domain naming — задача transport layer.

Следствия: независимость ядра, простое тестирование, простой headless режим.

### 2.3 Визуализатор — обязательная часть v1 [РЕШЕНО]

Собственный визуализатор, не RViz. Отдельный процесс, подключается через отдельный протокол. Формат snapshot/event/command — часть архитектуры.

### 2.4 Типизированные точки расширения [РЕШЕНО]

Расширение не через «плагины вообще», а через определённые роли. Каждая роль — свой контракт, свои права, своя точка входа. Произвольным плагинам нельзя ломать мир.

### 2.5 Мир меняется через ядро [РЕШЕНО]

Если модуль хочет изменить мир — он делает это через kernel command, а не напрямую. Только ядро видит всю картину, может проверять инварианты, может гарантировать согласованность.

# Часть II — Концептуальная модель

## 3. Модель мира — сущности

### 3.1 Три слоя системы [РЕШЕНО]

```
┌─────────────────────────────────────────────────────┐
│         Управляющий сервер / планировщик            │
│         (fleet management, распределение миссий)    │
└──────────────────────┬──────────────────────────────┘
                       │ любой протокол
┌──────────────────────▼──────────────────────────────┐
│         Стек робота: Nav2, MoveIt2, custom          │
│         (реальный код который едет на робот)        │
└──────────────────────┬──────────────────────────────┘
                       │ любой протокол
┌──────────────────────▼──────────────────────────────┐
│         S2 симуляция  ←——→  Реальный робот          │
└─────────────────────────────────────────────────────┘
```

Каждый слой заменяем независимо. Стек робота не знает — работает с симуляцией или с реальным роботом.

### 3.2 Типы сущностей [РЕШЕНО]

**Agent** — управляемая сущность (робот). Получает команды извне через транспорт, имеет сенсоры, публикует данные. Единственный тип, «видимый» снаружи. Подробная структура — раздел 4.

**Actor** — сущность с поведением, не управляемая извне. Пешеход, дверь, лифт, конвейер, шлагбаум. Имеет состояние и логику изменения (FSM). Ключевое: дверь и пешеход — один тип с разным поведением. Подробности — раздел 10.

Состоит из: поза, FSM (конечный автомат), коллизионный шейп (может меняться при смене состояния), визуальный примитив.

**Prop** — пассивный объект без поведения. Бочка, ящик, паллета. Не двигается сам, может быть подобран (attachment). Виден лидару.

Состоит из: поза, movable флаг, коллизионный шейп, визуальный примитив, properties (словарь), attachment info.

**Zone** — область пространства, не физический объект. Невидима для лидара. Реагирует на присутствие сущностей (триггер). Подробности — раздел 8.

## 4. Модель агента

### 4.1 Структура агента [РЕШЕНО]

```
Agent
├── Identity
│   ├── id (уникальный числовой)
│   ├── name (строка)
│   └── domain_id (для транспорта)
│
├── Spatial
│   ├── world_pose (x, y, z, roll, pitch, yaw)
│   ├── world_velocity
│   └── kinematic_tree (дерево фреймов)
│
├── Shared State
│   ├── Single-owner fields (battery_level, held_objects, ...)
│   ├── Contributions (speed_scale от батареи, от зоны, от эффекта, ...)
│   └── Effective/Resolved values (effective_speed_scale, motion_locked, ...)
│
├── Capabilities (набор строковых тегов для matching эффектов)
│
├── Modules
│   ├── Actuation (locomotion, joints)
│   ├── Sensor (lidar, gnss, odometry, semantic detectors)
│   ├── Interaction (grabber, door_opener, elevator_user)
│   └── Resource (battery, faults, safety_mode)
│
├── Active Effects
│   ├── Own effects (из конфига агента, активны всегда)
│   └── Zone effects (от зон, активны пока агент в зоне)
│
├── Collision
│   ├── Bounding shape (capsule/box/sphere — грубый, для зон и коллизий)
│   └── Per-link shapes (точные, для raycast)
│
├── Visual (тип + цвет для визуализатора)
│
└── Attachment info (привязан ли к актору, например к кабине лифта)
```

### 4.2 Capabilities — теги совместимости [РЕШЕНО]

Набор строковых тегов, объявляемых модулями при инициализации. Используются для matching эффектов.

Примеры:
```
DiffDrive:     ["surface_contact", "wheeled", "2d_motion"]
Multicopter:   ["airborne", "3d_motion"]
Lidar2D:       ["lidar_sensor", "optical"]
GNSS:          ["gnss_sensor", "global_position"]
Battery:       ["has_battery"]
```

### 4.3 Минимальный состав агента [ОТКРЫТО]

Должен ли агент иметь хотя бы один Actuation модуль? Или допустим «наблюдающий» агент (только сенсоры)? Подробнее — вопрос Q4 в разделе 27.

## 5. Таксономия модулей

### 5.1 Четыре типа модулей [РЕШЕНО]

#### Actuation — как агент исполняет действия

Отвечает на вопрос: **«как агент двигается и исполняет команды?»**

- Locomotion (DiffDrive, Multicopter, Holonomic, PointGoal)
- Joint control (JointVelocity)

Контракт:
- Вход: команда от транспорта (через входной буфер), dt
- Читает: effective constraints из Shared State
- Выход: DesiredVelocity (vx, vy, vz, wx, wy, wz)
- Результат проходит через цепочку эффектов перед применением — единственный тип модуля с таким поведением
- Не делает: IO, аллокации, блокировки на горячем пути

#### Sensor — что агент наблюдает

Отвечает на вопрос: **«что агент видит/измеряет вовне?»**

- GNSS, Lidar2D, Odometry
- Semantic detectors (ArUco, pedestrian detection)

Контракт:
- Вход: snapshot агента, kinematic tree, активные сенсорные эффекты, dt
- Читает: состояние мира через World Query API
- Выход: сенсорные данные → в snapshot → транспорт публикует
- Имеет собственную частоту публикации (может быть реже тика)
- Не пишет в Shared State (это делают Resource модули)

#### Interaction — как агент инициирует взаимодействие с миром

Отвечает на вопрос: **«как агент взаимодействует с объектами/акторами?»**

- Grabber, DoorOpener, ElevatorUser, ArucoDetector, Receiver

Контракт:
- Вход: команда от стека робота (через входной буфер)
- Проверяет: условия через World Query API (зона, дистанция, состояние актора)
- Действие: Kernel Command (attach, detach) или событие в Event Bus (команда актору)
- Выход: результат → стеку робота через транспорт

#### Resource — какие у агента внутренние ресурсы и ограничения

Отвечает на вопрос: **«что робот сейчас может или не может делать и почему?»**

- Battery, Faults, Thermal, SafetyMode, EStop

Это не наблюдение внешнего мира и не действие. Это внутреннее состояние, которое ограничивает возможности.

Контракт:
- Ведёт single-owner state (battery_level, fault_flags)
- Публикует contributions в resolved fields (speed_scale, motion_locked)
- Публикует телеметрию в snapshot (BatteryState, DiagnosticState)
- Может читать Shared State (velocity — для расчёта drain)

Пример: Battery — хранит SOC, уменьшает каждый тик, если ниже порога — публикует contribution speed_scale=0.3, если ниже критического — contribution motion_locked=true. Actuation модуль читает resolved effective_speed_scale и effective_motion_locked, не зная откуда пришло ограничение.

### 5.2 Связь модулей [РЕШЕНО]

**Модули не знают друг о друге напрямую.** Запрещены прямые объектные ссылки, прямые вызовы методов чужих модулей, скрытые hard dependencies.

Все взаимодействия — через четыре канала (раздел 6). Battery не знает про DiffDrive. DiffDrive не знает про Battery. Оба работают с Shared State по своим контрактам.

## 6. Каналы коммуникации

### 6.1 Четыре канала [РЕШЕНО]

#### Shared State

Устойчивое состояние на агенте. Single-owner fields + contributions + resolved effective values. Читается/пишется модулями по контракту. Подробности — раздел 7.

Примеры: battery_level, held_objects, effective_speed_scale, motion_locked.

#### Event Bus (SimBus)

Одноразовые факты. Публикация → подписка. Синхронный dispatch внутри тика — по заранее собранному списку подписчиков, без аллокаций.

Примеры событий:
- AgentEnteredZone / AgentExitedZone
- DoorCommand / DoorStateChanged
- ElevatorCommand / ElevatorStateChanged
- ObjectAttached / ObjectReleased
- ObjectMoved
- WorldEffectToggled
- AgentCollision

#### World Query API

Read-only запросы к ядру. Без побочных эффектов, можно вызывать в любой момент тика.

Примеры:
- «В какой зоне я?»
- «Есть ли объект в радиусе R?»
- «Брось 360 лучей и скажи что задели» (raycast)
- «Где находится актор door_1?»
- «Какое состояние у актора elevator_1?»

#### Kernel Commands

Write-запросы к ядру. Ядро валидирует и исполняет. Может отклонить.

Примеры:
- AttachObject(object_id, agent_id, link_name)
- DetachObject(object_id)
- TransferObject(from_agent, to_agent, object_id)
- SendActorCommand(actor_id, command)
- TeleportAgent(agent_id, pose)
- SpawnProp / RemoveProp
- ToggleZone(zone_id, enabled)

### 6.2 Входные команды от транспорта

Входящие команды от внешнего мира (cmd_vel, open_door, grab) — не отдельный канал. Это деталь реализации буферизации между потоком транспорта и потоком симуляции.

Транспортный поток получает команду из ROS/WebSocket, кладёт в буфер модуля (под мьютексом). Симуляционный поток забирает из буфера в начале тика. Модуль видит «пришла команда», ему всё равно откуда.

### 6.3 Производительность каналов

Всё работает в одном потоке (кроме буферизации от транспорта). Shared State — прямое чтение памяти. Event Bus — вектор подписчиков, синхронный вызов. World Query и Kernel Commands — вызов функции ядра. Стоимость — один виртуальный вызов на операцию, наносекунды.

## 7. Shared State и Contribution/Resolver

### 7.1 Три класса состояния [РЕШЕНО]

#### Single-owner state

Одно поле — один писатель. Простая семантика: прочитал → обновил → записал.

Примеры: battery_level (пишет BatteryResource), held_objects (пишет Grabber), world_pose (пишет ядро).

#### Contribution state

Несколько источников вносят вклады, но не пишут итог напрямую. Каждый contribution — именованный, с указанием источника.

Примеры contributions к speed_scale:
- battery: 0.85 (от BatteryResource)
- ice_zone: 0.20 (от зонального эффекта IceModifier)
- boost_zone: 1.15 (от зонального эффекта BoostModifier)
- overweight: 0.80 (от PayloadResource при 5 бочках)

#### Effective/Resolved state

Итог, вычисляемый resolver-ом из contributions. Читается Actuation и другими потребителями.

Примеры: effective_speed_scale, effective_motion_locked, effective_velocity_addition.

### 7.2 Merge rules [ПРЕДВАРИТЕЛЬНО]

**Scale contributions — multiplicative:**
```
effective_speed_scale = clamp(product(all_scale_contributions), 0.0, max)
```
Пример: 0.85 × 0.20 × 1.15 × 0.80 = 0.156

**Additive motion — отдельный канал:**
```
effective_velocity_addition = sum(all_additive_contributions)
```
Пример: конвейер (+1.0 по X) + ветер (+0.5 по Y) = (+1.0, +0.5, 0.0)

**Enable/disable flags — AND:**
```
effective_motion_locked = OR(any contribution says locked)
```
Если хоть один источник (e-stop, battery critical, зональный lock) говорит «стоп» — стоим.

### 7.3 Resolver как фаза тика [РЕШЕНО]

Resolver — явная фаза в тиковой петле. Запускается после всех contributions, перед Actuation.

```
1. Resource модули публикуют contributions
2. Зональные эффекты (CONTINUOUS) публикуют contributions
3. ──── RESOLVER ────
   effective_speed_scale = clamp(product(scales), 0, max)
   effective_motion_locked = OR(locks)
   effective_velocity_addition = sum(additions)
4. Actuation читает effective values
```

Один проход по массиву contributions, наносекунды. Порядок явный, неоднозначности нет.

### 7.4 Стандартные resolved fields для v1 [ОТКРЫТО]

Предварительный список:
- effective_speed_scale (double)
- effective_motion_locked (bool)
- effective_velocity_addition (Vec3)

Нужно ли: effective_manipulation_locked, effective_sensor_degraded, effective_max_speed_cap?

Подробнее — вопрос Q5 в разделе 27.

### 7.5 Пример потока данных

```
BatteryResource:     contribution(speed_scale, 0.85, source="battery")
IceModifier (зона):  contribution(speed_scale, 0.20, source="ice_zone_3")
EStopResource:       contribution(motion_locked, true, source="estop")

      ↓ RESOLVER ↓

effective_speed_scale = 0.85 × 0.20 = 0.17
effective_motion_locked = true  (e-stop active)

      ↓

DiffDrive Actuation: читает effective_motion_locked → true → velocity = 0
```

BatteryResource не знает про IceModifier. DiffDrive не знает откуда пришёл lock. Визуализатор может показать все contributions с источниками — оператор видит «почему робот стоит».

# Часть III — Механики мира

## 8. Зоны и триггеры

### 8.1 Концепция [РЕШЕНО]

Зона — область пространства, которая реагирует на присутствие сущностей. Невидима для лидара, не физический объект. Зона — это триггер: сообщает «сущность X вошла / вышла», а реакция — отдельная вещь (эффект, модуль, актор).

### 8.2 Унифицированная модель взаимодействий [РЕШЕНО]

Все взаимодействия в симуляции сводятся к схеме:

```
Триггер → Условие → Реакция
```

| Use case | Триггер | Условие | Реакция |
|----------|---------|---------|---------|
| Лёд | Агент в зоне ice_patch | capability: surface_contact | Contribution speed_scale каждый тик |
| ArUco маркер | Агент в зоне aruco_42 | capability: aruco_detection | Publish позиции маркера агенту |
| Дверь | Команда open_door + агент в trigger_zone | Дверь closed | Actor FSM: closed → opening → open |
| Лифт | Команда call_elevator + агент в зоне вызова | — | Actor FSM меняет состояние |
| Зарядка | Агент в зоне charging | capability: has_battery | Contribution: увеличение battery_level |
| Туман | Глобальная зона (infinite), enabled=true | capability: optical | Sensor effect: уменьшить range |
| Прокол | Агент вошёл в puncture zone | capability: wheeled | Mutation: tire_ok = false (необратимо) |
| Батарея | Каждый тик (resource module) | — | Contribution speed_scale от уровня |
| Конвейер | Агент в зоне конвейера | — | Contribution velocity_addition |
| Перегруз | 5 бочек загружено | — | Contribution speed_scale от payload |

### 8.3 Форма зон (v1) [РЕШЕНО]

- **Sphere** — центр + радиус
- **AABB** — ориентированный по осям параллелепипед
- **Infinite** — весь мир (для глобальных эффектов: туман, ветер)

### 8.4 Привязка зон [РЕШЕНО]

- **Fixed** — фиксированная позиция в мире
- **Attached to Agent** — следует за агентом (или конкретным link-ом)
- **Attached to Actor** — следует за актором

### 8.5 Переключение в рантайме [РЕШЕНО]

Зона имеет флаг `enabled`. Переключается через Kernel Command (из визуализатора или через API). При выключении — все внутри считаются «вышедшими», эффекты снимаются.

### 8.6 Что зоны отслеживают [РЕШЕНО для v1]

Агентов (всегда). Объекты (опционально, по флагу).

## 9. Система эффектов

### 9.1 Четыре типа эффектов [РЕШЕНО]

**MODIFIER** — модифицирует движение через contribution каждый тик. Публикует speed_scale или velocity_addition, resolver собирает итог.

Примеры: лёд (scale 0.2), ветер (additive velocity), конвейер (additive velocity), boost (scale 1.15).

**CONTINUOUS** — каждый тик изменяет single-owner field в Shared State. Работает пока агент в зоне (или всегда, если собственный эффект).

Примеры: зарядная станция (battery_level += charge_rate × dt).

**MUTATION** — одноразовое необратимое воздействие при входе в зону. Не снимается при выходе.

Примеры: прокол колеса (tire_ok = false).

**SENSOR** — модифицирует параметры сенсорных модулей. Применяется перед генерацией сенсорных данных.

Примеры: туман (max_range × 0.6), EM-помехи (noise_std × 10.0).

### 9.2 Зональные vs собственные эффекты [РЕШЕНО]

**Зональный** — от зоны. Активируется при входе, снимается при выходе (кроме MUTATION).

**Собственный (агентный)** — задан в конфиге агента. Активен с момента старта, не снимается. Примеры: BatteryModifier, PayloadModifier.

### 9.3 Capabilities matching [РЕШЕНО]

Эффект объявляет `required_capabilities`. Агент объявляет capabilities через модули. Эффект применяется только при полном совпадении. Проверка — один раз при входе в зону, результат кешируется.

```
IceModifier     требует: ["surface_contact"]  → дрон не скользит
WindModifier    требует: ["airborne"]          → колёсный не сдувается
FogSensorEffect требует: ["optical"]           → GNSS не затрагивается
```

### 9.4 Как MODIFIER работает с Contribution/Resolver [ПРЕДВАРИТЕЛЬНО]

MODIFIER не модифицирует velocity напрямую. Он публикует contribution:

```
IceModifier:   contribution(speed_scale, 0.2, source="ice_zone_3")
WindModifier:  contribution(velocity_addition, Vec3(0.5, 0, 0), source="wind_zone")
```

Resolver собирает все contributions (от эффектов + от resource модулей) и вычисляет effective values. Actuation читает effective values.

Старая модель «суммирование дельт» заменена на contribution/resolver.

## 10. Акторы — конечные автоматы

### 10.1 Концепция [РЕШЕНО]

Все интерактивные объекты с поведением — акторы с FSM. Единый паттерн: набор состояний, переходы (по событию / по таймеру), действия при переходах.

### 10.2 Примеры

**Дверь:**
```
states: [closed, opening, open, closing]
transitions:
  closed → opening:  по команде OPEN от агента в trigger_zone
  opening → open:    через 2 сек
  open → closing:    через 10 сек без агента / по команде CLOSE
  closing → closed:  через 2 сек
actions:
  on_enter(open):    убрать коллизионный шейп
  on_enter(closed):  вернуть коллизионный шейп
events:
  каждый переход:    DoorStateChanged{id, new_state}
```

**Лифт:**
```
states: [idle_N, moving_to_N, doors_opening_N, ready_N, doors_closing_N]
transitions:
  idle_1 → moving_to_3:       по команде call(3)
  moving_to_3 → doors_opening_3:  через T сек
  doors_opening_3 → ready_3:      через 2 сек
  ready_3 → doors_closing_3:      по команде go(1) от агента внутри
actions:
  on_enter(moving):  закрыть двери, перемещать кабину + attached агентов
  on_enter(ready):   открыть двери, событие elevator_ready
```

**Пешеход:**
```
states: [walking, waiting]
transitions:
  walking → waiting:  достиг waypoint
  waiting → walking:  через pause_duration
behavior:
  while(walking):     двигаться к waypoint со скоростью speed
  on_enter(waiting):  выбрать следующий waypoint
```

### 10.3 Акторы и коллизии [РЕШЕНО на уровне принципа]

Актор может менять свой коллизионный шейп при смене состояния (дверь открыта — шейп убран, закрыта — есть). Это действие при переходе FSM, исполняется через ядро.

## 11. Attachment — механизм привязки

### 11.1 Концепция [РЕШЕНО]

Attachment — связь «ребёнок следует за родителем». Каждый тик ядро обновляет позы привязанных: поза = поза_родителя + offset.

### 11.2 Что покрывает

**Prop в захвате робота:** Prop привязан к link агента. Робот двигается → prop следует. При release — привязка удаляется.

**Агент в лифте:** Agent привязан к Actor «elevator_cabin». Кабина движется → агент следует. Движение робота (cmd_vel) — в локальной системе кабины. Мировая поза = поза_кабины + локальная_поза.

### 11.3 Что НЕ покрывает — используются зональные эффекты

**Конвейер:** робот не жёстко привязан — может съехать. Конвейер — зональный MODIFIER, добавляет contribution velocity_addition. Скорости складываются. Смежные конвейеры: выход из одного → вход в другой в пределах тика.

### 11.4 Правила [РЕШЕНО]

- Только определённые пары: Prop → Agent link, Agent → Actor
- Создание/удаление — только через Kernel Command (не напрямую модулем)
- При удалении родителя — все привязанные отвязываются

### 11.5 Передача объекта между роботами [РЕШЕНО]

Через ядро, потому что операция затрагивает двух агентов:

```
Робот B → Kernel Command: TransferObject(from=A, to=B, obj=box_1)
Ядро проверяет: оба в transfer_zone, A имеет объект, B имеет место
Ядро: detach от A, attach к B, обновить Shared State обоих
События: ObjectReleased (A), ObjectReceived (B)
```

## 12. Взаимодействие модулей с миром

### 12.1 Полная цепочка: открытие двери [РЕШЕНО]

```
Стек робота                          Симуляция
                                     
1. "open_door(door_1)"               DoorOpener (Interaction модуль)
   ──── транспорт ───────────→        │
                                     2. World Query: "я в trigger_zone двери?"
                                     │
                                     3. Да → Event Bus: DoorCommand{door_1, OPEN}
                                     │
                                     4. Actor "door_1" получает событие
                                     │  FSM: closed → opening → (2 сек) → open
                                     │
                                     5. Event Bus: DoorStateChanged{door_1, OPEN}
                                     │
                                     6. DoorOpener получает ответ → результат
                                     │
7. "door opened"  ←── транспорт ──   результат в snapshot
```

DoorOpener не знает как дверь работает. Дверь не знает кто открывает. Связь только через Event Bus. Если актор запрограммирован на отказ — DoorOpener передаёт отказ стеку.

### 12.2 Полная цепочка: захват объекта [РЕШЕНО для v1]

```
1. Стек: "grab(box_1)" → Grabber (Interaction модуль)
2. World Query: "объект box_1 в радиусе?"
3. Kernel Command: AttachObject(box_1, robot_0, gripper_link)
4. Ядро валидирует и исполняет
5. Event Bus: ObjectAttached{box_1, robot_0}
6. Grabber обновляет held_objects в Shared State
7. Результат → стеку
```

Без физики захвата, без contact solver. Просто: запрос → проверка → attachment → результат.

### 12.3 Полная цепочка: payload и скорость

```
1. 5 бочек загружены (held_objects.size == 5)
2. PayloadResource (Resource модуль): 
   - читает held_objects
   - публикует contribution(speed_scale, 0.80, source="payload")
3. BatteryResource: contribution(speed_scale, 0.85, source="battery")
4. RESOLVER: effective_speed_scale = 0.80 × 0.85 = 0.68
5. DiffDrive (Actuation): desired_velocity × 0.68
6. Стек робота видит скорость робота 68% от нормальной
7. Sensor публикует состояние payload → стек знает: 5 бочек загружено
8. Стек: "5 бочек нельзя" → ставит робота в ошибку (программное ограничение)
```

Симуляция честно замедлила робота (физическое ограничение). Стек сам решает что 5 — нельзя (программное ограничение). Разделение ответственности.

## 14. Kinematic Tree

### 14.1 Назначение [РЕШЕНО]

Дерево связанных частей тела робота. База → лидар, рука и т.д. Каждая часть — положение относительно родителя.

### 14.2 Два источника [РЕШЕНО]

- **URDF** — парсится `urdfdom` (без ROS зависимости)
- **YAML** — упрощённый формат для простых агентов

### 14.3 Типы джоинтов (v1) [РЕШЕНО]

revolute, prismatic, fixed, continuous (как в URDF).

### 14.4 Обновление [РЕШЕНО]

Каждый тик после Joint модулей: обновить все трансформы из углов джоинтов + базовая поза. Per-link коллизионные шейпы пересчитываются автоматически.

### 14.5 Виртуальные линки [РЕШЕНО]

Сенсорный модуль может добавить виртуальный link к дереву (offset от базы) без записи в URDF.

### 14.6 TF публикация [РЕШЕНО]

Полное дерево с привязкой к точке спавна:

```
earth → map   (статичный, 1 Гц, ECEF позиция точки спавна робота)
map   → odom  (статичный, identity — odom = map в точке спавна)
odom  → base_link (динамический, @ transport_rate — смещение относительно старта)
```

Робот всегда стартует в `(0, 0, 0)` odom-фрейма. `earth→map` уникален для каждого домена — определяет GPS-позицию точки спавна. Локализация — идеальная (нет дрейфа odom). Nav2 видит корректный `/tf` с нулевым `map→odom` offset.

## 15. Коллизии и Raycast

### 15.1 Три слоя геометрии для Raycast [РЕШЕНО]

**Статический BVH:** геометрия мира. Строится при загрузке, не перестраивается.

**Динамический BVH:** movable props. Перестраивается по событию ObjectMoved (не каждый тик).

**Capsule/Sphere list:** агенты и акторы. Обновляется каждый тик. Аналитическая проверка. 100 агентов = 100 проверок на луч.

### 15.2 Коллизии [ПРЕДВАРИТЕЛЬНО]

**Агент + статика:** откат позиции.

**Агент + агент:** детерминированный приоритет по AgentId. Агент с меньшим id не откатывается.

Известная проблема: каскадные откаты при N агентах. См. P1 в разделе 25.

## 18. Тройная буферизация

### 18.1 Принцип [РЕШЕНО]

Три буфера: writer (пишет ядро), ready (готов для чтения), reader (читает транспорт). Writer и reader никогда не на одном буфере. Атомарные swap без локов.

### 18.2 Ограничение [ОТКРЫТО]

Корректно для одного writer и одного reader. Несколько транспортов одновременно — нужен один reader-поток или другая схема. Подробнее — P3 в разделе 25.

## 19. Тиковая петля — порядок обработки

### 19.1 Полный порядок [ПРЕДВАРИТЕЛЬНО]

```
1. Обновить акторов
   - FSM переходы, таймеры
   - Обновить позы акторов

2. Обновить зоны
   - Кто вошёл, кто вышел → события
   - Применить/снять зональные эффекты
   - Применить MUTATION для новых вошедших (одноразово)

3. Для каждого агента:

   a. Resource модули
      - BatteryResource: drain, publish contribution(speed_scale)
      - PayloadResource: check held_objects, publish contribution(speed_scale)
      - FaultResource: check faults, publish contribution(motion_locked)

   b. Собственные эффекты (CONTINUOUS)
      - BatteryModifier: contribution(speed_scale) по уровню батареи

   c. Зональные эффекты (CONTINUOUS)
      - ChargingEffect: battery_level += charge_rate × dt
      - Зональные MODIFIER: contribution(speed_scale), contribution(velocity_addition)

   d. ──── RESOLVER ────
      - effective_speed_scale = clamp(product(scales))
      - effective_motion_locked = OR(locks)
      - effective_velocity_addition = sum(additions)

   e. Actuation модули → desired_velocity
      - Читает effective constraints
      - Возвращает DesiredVelocity

   f. Применить effective_velocity_addition (конвейер, ветер)
      - desired += effective_velocity_addition

   g. Кинематика → новая поза
   h. Surface snap (z из heightmap, если surface_contact)
   i. Коллизии → откат если нужно

   j. Joint модули → обновить углы
   k. Обновить Kinematic Tree

   l. Sensor модули → сгенерировать данные
      - Применить SENSOR эффекты перед генерацией

   m. Interaction модули → обработать входящие команды

4. Обновить attachments
   - Привязанные props/агенты → пересчитать позы

5. Snapshot → тройной буфер

6. Публикация состояния мира визуализатору
   (с собственной частотой, не каждый тик)

7. Транспорт spin
```

### 19.2 Обоснование порядка

**ZoneSystem перед Actuation:** эффект применяется в том же тике когда агент вошёл в зону (без задержки в 1 тик).

**Resource и CONTINUOUS перед Resolver:** все contributions собраны до вычисления effective values.

**Resolver перед Actuation:** Actuation читает готовые effective values, не сырые contributions.

**Sensors после кинематики:** сенсоры видят актуальную позу текущего тика.

**Interactions последние:** обработка команд может генерировать Kernel Commands, которые применятся в начале следующего тика.

# Часть VI — Внешний мир

## 20. Транспортная модель

### 20.1 Принцип: транспорт = маппинг [РЕШЕНО]

Сенсорный плагин генерирует данные и кладёт в `SharedState` агента. `SimTransportBridge` собирает кадр `AgentSensorFrame` (только сенсоры с новым `seq`) и передаёт в `ITransportAdapter`. Сенсор не знает про ROS/топики. Транспорт не знает как данные получены.

```
IAgentPlugin (gnss/imu/diff_drive)
  └─ данные → SharedState агента

SimTransportBridge::on_post_tick()  (частота transport_rate, по умолчанию 30 Гц)
  └─ per agent: AgentSensorFrame (world_pose + только новые сенсоры)
       └─ ITransportAdapter::publish_agent_frame(frame)
            └─ Ros2TransportAdapter:
                 ├─ /gnss/fix  (sensor_msgs/NavSatFix)
                 ├─ /imu/data  (sensor_msgs/Imu)
                 ├─ /odom      (nav_msgs/Odometry)
                 └─ /tf        (odom→base_link, earth→map, map→odom)
```

### 20.2 Изоляция агентов [РЕШЕНО]

Каждый агент — изолированный «робот». В ROS: отдельный domain ID. Nav2 с `ROS_DOMAIN_ID=N` видит чистые топики без префиксов. Один `rclcpp::Context` и `rclcpp::Node` на domain_id.

### 20.3 TF-дерево — привязка к точке спавна [РЕШЕНО]

Каждый робот получает `map`/`odom` фреймы, привязанные к его точке спавна:

```
earth → map  (ECEF позиция точки спавна, уникальная для каждого робота)
map   → odom (identity — odom совпадает с map в точке спавна)
odom  → base_link (смещение относительно старта, динамический TF @ transport_rate)
```

`earth→map` вычисляется: `ECEF_spawn = ECEF(geo_origin) + R_enu^T · (spawn_x, spawn_y, spawn_z)`.

Робот всегда стартует в `(0, 0, 0)` своего odom-фрейма. GNSS координаты в точке спавна соответствуют `geo_origin + ENU-смещение`.

### 20.4 Режимы работы [РЕШЕНО]

- **Без транспорта** — headless тесты (MockTransportAdapter)
- **ROS2** — `Ros2TransportAdapter`, каждый агент в своём домене
- **WebSocket** — JSON для визуализатора и не-ROS клиентов
- **ROS2 + WebSocket** — одновременно (разные адаптеры)

### 20.5 FastDDS и контейнеры [РЕШЕНО]

По умолчанию FastDDS на `network_mode: host` предпочитает shared memory (SHM). Это ломает связь с контейнерами, использующими только UDP. Оба контейнера должны использовать единый транспорт:

```xml
<type>UDPv4</type>
<useBuiltinTransports>false</useBuiltinTransports>
```

Конфиг монтируется через `FASTRTPS_DEFAULT_PROFILES_FILE=/root/.ros/fastdds.xml`.

# Часть VIII — Статус

## 25. Известные архитектурные проблемы

### P1 — Каскадные откаты при коллизиях агент-агент [ВЫСОКИЙ]

Приоритет по AgentId: агент с id=99 может быть выброшен через стену при последовательных откатах. Нужен двухпроходный алгоритм.

### P2 — Race condition во входных буферах модулей [ВЫСОКИЙ]

Транспортный поток пишет в буфер, симуляционный читает. Контракт «модуль обязан защищать буфер мьютексом» должен быть формализован.

### P3 — Тройная буферизация при нескольких reader-ах [СРЕДНИЙ]

Несколько транспортов читают одновременно — гонка. Нужен один reader-поток или другая схема.

### P4 — pluginlib зависит от ROS [СРЕДНИЙ]

pluginlib → ament_index, class_loader, rcpputils. Противоречие с «ROS-independent core». Нужно решение.

### P5 — Heightmap не поддерживает этажи [СРЕДНИЙ]

Одно z на (x, y). Нет рамп, мостов, многоэтажных зданий. Blocker для лифтов.

### P6 — Template virtual methods в ITransportNode [СРЕДНИЙ]

Невозможны в C++. Конкретное решение (CRTP, type erasure) не определено.

## 27. Открытые вопросы для обсуждения

Сгруппированы по темам. Порядок — по убыванию влияния на остальную архитектуру.

---

### Модель времени

**Q1. Fixed dt или variable dt?**
Fixed dt: каждый тик ровно 10 мс (при 100 Гц). Проще для детерминизма — одинаковый прогон каждый раз. Variable dt: реальное время между тиками, clamped. Ближе к реальности, но результат зависит от производительности машины.

**Q2. Pause / step / resume?**
Нужен ли пошаговый режим для отладки? Если да — нужна команда из визуализатора. Как это взаимодействует с транспортом (ROS стек ожидает непрерывную симуляцию)?

**Q3. Ускорение симуляции?**
Гнать быстрее реального времени (2x, 10x) для быстрого прогона тестов. Требует отделения sim_time от wall_time с самого начала. Влияет на транспорт (/clock).

---

### Модель агента

**Q4. Минимальный обязательный состав агента?**
Должен ли агент иметь хотя бы один Actuation модуль? Допустим ли «наблюдатель» (только сенсоры)? Допустим ли агент без сенсоров (только actuation)?

**Q5. Полный список resolved fields для v1?**
Предварительно: effective_speed_scale, effective_motion_locked, effective_velocity_addition. Нужно ли: effective_manipulation_locked, effective_max_speed_cap, effective_sensor_degraded?

**Q6. Typed vs dynamic Shared State?**
Фиксированная схема полей (struct) или произвольный словарь (map)? Typed: быстрее, безопаснее, проверка при компиляции. Dynamic: гибче, плагины добавляют свои поля. Или гибрид: typed core + dynamic plugin fields?

---

### Зоны и эффекты

**Q7. Типизация зон?**
Все зоны одинаковые (триггер + список эффектов)? Или подтипы: effect zone (лёд), trigger zone (дверь), interaction zone (передача объектов)? Подтипы яснее семантически, одинаковые — проще.

**Q8. Overlapping zones?**
Агент в зоне льда и тумана одновременно. Обе генерируют contributions — все попадают в resolver? Конфликты? Что если зоны противоречат (одна ускоряет, другая замедляет — resolver разруливает через multiply)?

**Q9. Как SENSOR эффекты вписываются в contribution модель?**
MODIFIER → contributions → resolver → effective_speed_scale. Логично. Но SENSOR эффекты (туман → max_range × 0.6) модифицируют параметры конкретного сенсора, а не общий resolved field. Нужен ли отдельный resolver для сенсоров? Или SENSOR эффекты остаются прямой модификацией?

**Q10. Послесвечение CONTINUOUS?**
Должен ли CONTINUOUS эффект работать N секунд после выхода из зоны? Пример: зона заражения — даже выехав, агент «болеет» ещё 30 секунд. Или это моделируется как MUTATION + отдельный таймер?

### Акторы

**Q11. Декларативный FSM или программный?**
YAML (описание состояний и переходов в конфиге — проще для пользователей) или плагин (каждый актор — код, гибче)? Комбинация: базовые акторы (дверь) декларативно, сложные (лифт) — плагином?

**Q12. Spatial behavior пешехода?**
Пешеход обходит стены или ходит сквозь (строго по waypoints)? Обход требует навигации для акторов — большой scope. Waypoints — проще, но менее реалистично.

**Q13. Многоэтажность и лифт?**
Лифт перемещает агента между этажами. Heightmap — одно z на (x,y). Как лифт работает? Несколько «плоскостей» на разных z? Отдельные зоны коллизий на каждом этаже?
