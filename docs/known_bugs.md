# Known Bugs

## [OPEN] Сильные задержки при включённых overlay-плагинах (траектория, путь)

**Симптомы:**
- При 3 включённых `trajectory_recorder` / `path_display` задержка drag-гизмо достигает десятков секунд и более
- Команды `diff_drive` (`plugin_input`) отрабатывают с задержкой 10–60 секунд
- Чем больше агентов с включёнными траекториями — тем хуже

**Диагностика:**
Каждый SSE-фрейм (30fps) содержит полный снапшот включая все точки траекторий
(до 300 точек × 3 координаты на агента). При 3 агентах — ~900 3D-точек в JSON каждые 33ms.
Отправка больших сообщений потенциально блокирует мьютексы, которые также удерживает
поток симуляции при `publish()`, и HTTP-handler при командах.

**Попытки решения (не помогли):**

1. **Throttle plugins_data**: включать данные плагинов только раз в 10 кадров (~3Hz вместо 30Hz).
   Изменён `snapshot_to_json()` — добавлен параметр `include_plugins`.
   В `handle_pending_snapshots()` счётчик `snap_counter_` определяет когда включать.

2. **MSG_DONTWAIT в send()**: заменить блокирующий `send()` на неблокирующий.
   Если буфер сокета занят — кадр дропается вместо блокировки потока.

3. **Освобождение snapshot_mutex_ до сериализации**: основная гипотеза —
   JSON-сериализация (медленная) происходила внутри `snapshot_mutex_`, блокируя
   поток симуляции. Исправлено: снапшот копируется под мьютексом, мьютекс отпускается,
   затем сериализация. Затронуто: `handle_pending_snapshots()`, `force_broadcast_latest()`,
   `send_snapshot_now()`.

4. **Вынесение handle_pending_snapshots() из accept-цикла**: accept-цикл конкурировал
   с SSE-потоками за обработку снапшотов, что могло блокировать приём команд.

5. **JS-кэш overlay-линий**: добавлен `overlayLineCache` — BufferGeometry Three.js
   не пересоздаётся если количество точек и последняя точка не изменились.

**Возможные направления дальнейшего исследования:**
- Замерить реальный размер SSE-сообщений и пропускную способность (`tcpdump` / `ss -i`)
- Проверить не является ли узким местом сам JS (`performance.now()` вокруг `updateScene`)
- Рассмотреть бинарный формат (MessagePack / protobuf) вместо JSON для траекторий
- Рассмотреть отдельный SSE-поток только для позиций агентов (лёгкий, 30fps)
  и отдельный polling-endpoint для overlay-данных (тяжёлый, 2-3fps)
- Ограничить `max_points` в `trajectory_recorder` до 50-100 точек пока проблема не решена
